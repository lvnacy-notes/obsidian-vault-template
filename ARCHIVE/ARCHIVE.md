# ARCHIVE

The ARCHIVE directory serves as the changelog and historical record for the Apparatus vault. It documents the evolution of the vault structure, configuration, and components over time.

## Purpose

This archive tracks:

- **Structural Changes** - Modifications to vault organization, folder hierarchies, and naming conventions
- **Configuration Updates** - Changes to Obsidian settings, plugin configurations, and workspace preferences
- **Component Additions** - New plugins, CSS snippets, templates, or modules installed and integrated
- **Workflow Refinements** - Improvements to processes, automation scripts, and operational procedures
- **Documentation Updates** - Changes to guidelines, references, and internal documentation
- **Module Evolution** - Updates and versions of installed submodules and their functionality

## Organization

Each changelog entry should document:

- **Date** - When the change occurred
- **Category** - The type of change (structure, configuration, plugins, templates, scripts, documentation, or modules)
- **Description** - Clear explanation of what changed and why
- **Impact** - How this change affects the vault or your workflow
- **Implementation Details** - Any configuration steps, file modifications, or technical notes

## Entry Format

Use a consistent format for new entries:

```markdown
## [Date] - [Category]

### Summary
Brief description of the change.

### Details
Detailed explanation of what changed, why, and how.

### Impact
How this affects the vault, workflows, or modules.

### Technical Notes
Any configuration files affected, scripts modified, or setup steps required.
```

## Categories

- **Structure** - Folder organization, naming conventions, navigation hierarchy
- **Configuration** - Plugin settings, Obsidian core settings, workspace preferences, theme customization
- **Plugins** - Installation, removal, or significant updates to community or core plugins
- **Templates** - New templates, modifications, or template system improvements
- **Scripts** - New automation, script modifications, or enhancements to Templater functionality
- **Documentation** - README updates, reference materials, cheatsheets, guidelines
- **Modules** - Submodule additions, updates, configuration, or integration changes
- **Styling** - CSS snippet additions, modifications, or theme-related changes
- **Maintenance** - Cleanup, optimization, dependency updates, or performance improvements

## Best Practices

1. **Regular Updates** - Document changes shortly after they occur to maintain accuracy
2. **Clarity** - Write clearly enough that someone unfamiliar with the change can understand it
3. **Context** - Explain not just *what* changed, but *why* it changed
4. **Links** - Reference affected files, modules, or documentation using proper links
5. **Versioning** - For major vault updates, consider creating versioned changelog sections
6. **Impact Assessment** - Be explicit about whether a change is breaking, non-breaking, or optional

## Version Control

This archive works in tandem with Git commit history:

- **Git commits** handle technical implementation details
- **ARCHIVE entries** provide high-level, human-readable context
- **Together** they create a complete record of vault evolution
